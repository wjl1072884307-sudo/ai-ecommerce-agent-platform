# V1.1 会话中心产品模型确认

本文档对应 `TASKS_ONLINE_V1.1` 的 Phase 1，用于确认会话中心的产品定位、数据边界和后续实现方向。Phase 1 只做产品模型与架构边界确认，不改业务代码。

## 1. 产品定位

当前 Sessions 页面更接近 Agent 调试台：后台人员需要手动创建会话，再输入首条客户消息触发 Agent。这适合演示 Pipeline，但不符合真实客服系统。

V1.1 的目标是把它升级为“会话中心”：

- 客户发送第一条消息后自动创建会话。
- 系统自动识别售前、售后、物流、投诉、发票等类型。
- Agent 回复进入真实聊天记录。
- 右侧不再重复展示回复正文，而是展示上下文与决策依据。
- 匿名访客、未购买客户、已购买客户都能进入同一会话入口。

## 2. 数据库判断

### 2.1 不新增第二个物理数据库

V1.1 不建议为会话中心新增第二个物理数据库。

理由：

- 会话、消息、订单、商品、Agent Run、审核任务、工单、审计日志之间存在强关联。
- 售后场景通常需要在一次流程中同时写入消息、Agent Run、回复、审核任务、工单和审计记录，放在同一数据库内更容易保证一致性。
- 当前项目目标是在线 MVP 到 ToB 雏形，不是大规模独立 IM 平台。
- 额外数据库会带来跨库查询、事务一致性、备份、迁移、部署和排查成本。

结论：

- 保持单一业务数据库。
- 本地继续支持 SQLite。
- 部署继续使用 PostgreSQL。
- 如后续进入高并发 IM 或多渠道消息平台阶段，再评估独立消息库、搜索索引或事件流。

### 2.2 customer_id 不作为会话主键

`customer_id` 不应该作为会话或聊天内容主键。

理由：

- 一个客户可能有多次会话。
- 一个客户可能同时关联多个订单、商品和售后工单。
- 匿名访客、未登录客户、未购买客户没有稳定 customer_id。
- 主键应该标识记录自身，而不是业务参与者。

推荐边界：

- `customers.id` 标识客户。
- `sessions.id` 或未来 `conversations.id` 标识一次会话。
- `messages.id` 标识一条消息。
- `sessions.customer_id` 是可空外键，不是主键。
- 匿名访客通过 `visitor_id` 或 `visitor_token` 进入会话。

## 3. 统一入口设计

真实业务入口不应要求后台人员先创建会话。

推荐流程：

1. 客户从 Web、App、后台模拟入口或 API 发送第一条消息。
2. 后端判断请求是否携带 `session_id`。
3. 如果没有 `session_id`，自动创建会话。
4. 如果有 `customer_id`，绑定客户。
5. 如果没有 `customer_id`，生成或接收 `visitor_id`，创建匿名访客会话。
6. 保存客户消息。
7. Agent 判断会话类型、意图、订单上下文和风险。
8. Agent 生成回复并写入消息表。
9. 必要时创建审核任务或工单。
10. 返回会话、消息和决策上下文。

## 4. 售前与售后边界

会话中心不应拆成两个完全割裂的页面，但应支持明确分类。

推荐会话大类：

- `pre_sales`：售前咨询，例如价格、库存、规格、优惠、兼容性、发货时间。
- `after_sales`：售后服务，例如退货、退款、换货、质保、商品质量问题。
- `logistics`：物流问题，例如发货、配送、签收、物流异常。
- `complaint`：投诉或强负面情绪。
- `invoice`：发票相关问题。
- `other`：无法明确分类的问题。

推荐实现方式：

- 同一个会话中心页面。
- 左侧使用类型筛选和状态筛选。
- Agent 自动写入 `conversation_type` 和 `intent`。
- 售前问题不强行进入订单售后链路。
- 售后问题优先尝试绑定订单。

## 5. 匿名访客与未购买客户

匿名或无购买记录客户仍然可以咨询。

处理原则：

- 售前问题可以正常回答。
- 售后问题不能直接承诺退款、退货、换货或赔付。
- Agent 应引导客户补充订单号、手机号、购买账号、商品名称、购买时间、问题照片或视频。
- Demo 场景可以 mock 回复，但必须明确这是“需要补充订单信息”的处理。

示例行为：

- 匿名客户问“这款耳机有降噪吗？”：按售前咨询回答。
- 匿名客户问“我的耳机坏了能退吗？”：提示需要订单信息和问题凭证。
- 已购客户问“耳机开箱有杂音需要退换”：查询订单、判断售后规则、生成回复，必要时创建审核任务和工单。

## 6. 页面结构确认

### 6.1 左侧：会话队列

左侧应从普通列表升级为会话队列。

筛选项：

- 全部
- 售前
- 售后
- 物流
- 投诉
- 发票
- 待人工
- 已关闭

搜索项：

- 客户名称
- customer_id
- visitor_id
- 订单号
- 商品名
- 会话标题
- 会话摘要

会话卡片应展示：

- 会话摘要或标题
- 会话类型
- 客户或访客标识
- 订单绑定状态
- 处理状态
- 最新消息时间
- 是否需要人工介入

### 6.2 中间：真实聊天窗口

中间只负责展示聊天内容：

- 客户消息
- Agent 回复
- 人工客服回复
- 必要的系统提示

要求：

- Agent 回复必须持久化到消息表。
- Agent 回复语言跟随客户最近一条消息语言。
- UI 中英文切换不翻译历史聊天内容。
- Enter 发送，Shift + Enter 换行。

### 6.3 右侧：上下文与决策面板

右侧不再重复展示 Agent 回复正文。

右侧应展示：

- 客户信息：customer_id、visitor_id、是否匿名、是否已购买。
- 订单上下文：订单号、商品、支付状态、物流状态、售后状态。
- Agent 判断：会话类型、意图、置信度、风险等级、是否需要人工。
- RAG 来源：命中文档、命中片段、来源摘要。
- 审核与工单：审核任务状态、工单号、工单类型、工单状态、负责人。
- 技术诊断：Agent Run ID、LLM Provider、是否 fallback、节点日志入口。

## 7. 新建会话入口定位

“新建会话”不应作为真实业务主流程。

V1.1 建议将其降级为：

- “模拟客户咨询”
- 或“创建测试会话”

测试入口可选择：

- 匿名访客
- 已购客户
- 售前咨询
- 售后问题
- 绑定订单
- 不绑定订单

真实主流程应由客户发送第一条消息自动创建会话。

## 8. Phase 1 验收结论

Phase 1 的产品模型确认如下：

- 会话中心不是 Agent 调试台。
- 不新增第二个物理数据库。
- `customer_id` 不是会话主键。
- 匿名访客可进入咨询流程。
- 售前和售后在同一会话中心中分类管理。
- 中间聊天窗口展示真实消息。
- 右侧面板定位为上下文与决策依据。
- 手动新建会话降级为测试入口。
